# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository Nature

A documentation repository with a minimal zero-dependency Node.js build pipeline. It publishes one reference document as Markdown (canonical) plus two generated artifacts: a styled standalone HTML page and a machine-readable JSON export.

## Source of Truth

`agentic-ai-complete-reference.md` is the **only** file a human edits for content. Everything else downstream is generated:

- `dist/index.html` — styled HTML (generated by `node scripts/build.js`, gitignored, deployed to GitHub Pages)
- `commands.json` — structured export of every command row (generated by `node scripts/export-commands.js`, committed)

Do NOT hand-edit `dist/index.html` or `commands.json`. If either needs a change, the fix belongs in the source Markdown or the generator script.

## Build Commands

```sh
node scripts/build.js            # MD → dist/index.html
node scripts/export-commands.js  # MD → commands.json
```

Both are plain Node.js with no dependencies — no `package.json`, no `npm install`. Run `node` ≥ 18.

## Document Structure

- **Part I — Claude Code: Practical Commands.** Named `###` sections (Session Management, Context & Information, Model & Mode Control, Configuration, Extensions, CLI Flags, Keyboard Shortcuts, Skills, etc.), each a table of `command → description` pairs.
- **Part II — Universal AI Agent Command Glossary.** Roman-numeraled `###` sections (`I.` through `XXI.`). Larger sections use `####` lettered subsections (`A.`, `B.`, ...).
- **Special Sections** (a third `##` group, marked `## ⚠️ Special Sections`) — continues the Roman-numeral sequence (XXII, XXIII) but uses **3-column tables** and has distinct visual treatment. Currently holds anti-patterns (`XXII`, failure modes with a Detection Signal column) and hidden patterns (`XXIII`, informal practices with a When-to-Use column). These entries are NOT commands to execute — they're failure modes to watch for and instinctive strategies to codify.

When adding a new numbered section (Part II or Special), you must also extend `PART_II_SECTIONS` in `scripts/build.js` (Roman numeral → class, icon, tip tone) AND add a matching `.s-<cls>` CSS rule in that file. The build throws `No metadata for section ...` otherwise — fail-fast, not silent. Never renumber existing entries (breaks anchor links and changelog references).

Part II §X contains an ASCII-art lifecycle diagram inside a fenced code block — deliberate visual element, preserved by the build as a styled `<pre>` block.

## Editorial Conventions

- **Part I** command names are backtick-quoted (`` `/compact` ``) — they are literal invocations.
- **Part II** command names are bolded (`**Spawn**`) — they are conceptual patterns, not literal commands.
- Aliases are inline in the description (`Aliases: /reset /new`). The exporter recognizes this form.
- Tip callouts use `> 💡 ...` — the only emoji used in-content.
- Section dividers are `---` on their own line.
- A literal `|` inside a table cell must be escaped as `\|` (standard Markdown) — e.g., `` `/fast [on\|off]` ``. Both generators handle the escape.
- **3-column tables** (used in Special Sections XXII, XXIII) are automatically rendered with a third `.cmd-extra` column. The JSON exporter uses the 3rd header cell (lowercased and snake_cased) as the key name — so `| ... | ... | Detection Signal |` becomes `detection_signal` in the JSON entry.
- The document carries a date line near the top (currently "April 2026"); bump only on substantive revisions.

## Build Script Notes

Both scripts parse the Markdown directly line-by-line — they are not generic markdown engines. If you change the MD in structurally new ways (new heading depth, non-table content, new callout syntax), the parser will likely need a matching change. Bugs typically surface as a silently-empty HTML section or a malformed `commands.json` entry — re-run the script and diff against the previous commit to catch regressions.

`scripts/build.js` reproduces a specific hand-crafted dark theme with per-section color coding (`s-session`, `s-foreman`, etc.), a filter-button nav, a floating right-rail TOC, and a client-side scroll-spy. When adding a new top-level Part II section, the script needs to know which color class to assign — check the section-to-class mapping inside the script.

## Deployment

GitHub Pages via `.github/workflows/pages.yml` — runs `node scripts/build.js` on every push to `main` and deploys `dist/`. See `.github/PAGES_SETUP.md` for the one-time Settings → Pages switch (must be done by a human; can't be automated).

## Changelog

`CHANGELOG.md` tracks content-level changes per commit. Update it when content is added, removed, or materially reorganized; not needed for build-script tweaks.
